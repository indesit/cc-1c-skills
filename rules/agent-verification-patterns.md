# Agent Verification Patterns

Паттерны проверки результата работы AI-агента. Главный - итеративная сверка покрытия (reconciliation loop).

Применяется: при разработке, где есть набор входов (файлы, строки, записи, форматы) и набор правил/кода, который должен их все покрывать. Типовые случаи: миграции, объектные мэппинги, парсеры разноформатных файлов, загрузки, трансформации данных, обмены.

---

## Принцип

После написания/правки кода - отдельной задачей поручить агенту сверить ВЕСЬ реестр входов с правилами в коде, выявить расхождения, исправить, повторить до нуля. Сверка - не часть написания кода, а самостоятельный проверочный проход.

Ошибка покрытия (вход не обрабатывается ни одним правилом, или обрабатывается неверным) дешевле ловится сплошной сверкой, чем ручной выборкой или тестом «на удачу».

---

## Паттерн: итеративная сверка покрытия (reconciliation loop)

### Когда применять

- На входе много элементов с вариативной структурой (N файлов разных форматов, M строк правил маппинга, K записей данных).
- Код/правила должны однозначно покрывать каждый элемент.
- Ручная сверка каждого элемента трудозатратна и подвержена ошибкам.

### Как

1. Зафиксировать реестр входов - явный список: файлы, значения, строки, форматы.
2. Поручить агенту: для каждого входа определить, каким правилом/профилем/веткой кода он обрабатывается.
3. Собрать таблицу «вход -> правило», выделить непокрытые и конфликтные (один вход подходит под несколько правил).
4. Исправить код/правила под выявленные расхождения.
5. Повторить сверку. Критерий завершения: 0 непокрытых, 0 конфликтов - полное и однозначное покрытие.

### Ключевое

- Сверять по фактическому содержимому, не по имени/метке. В кейсе-источнике формат файла определялся по шапке листа, а не по имени файла (имя могло вводить в заблуждение).
- Полный реестр, не выборка. Цель - явный счёт «N/N покрыто, 0 расхождений».
- Итеративность: первый проход почти всегда даёт расхождения, они сводятся к нулю за несколько циклов.
- Финальную приёмку на реальных данных выполняет человек. Агент готовит сверку, но не заменяет приёмку.

Источник идеи: практический кейс миграции большого набора XLSX-файлов через несколько профилей чтения - агент сверял все файлы с профилями в коде до нуля расхождений.

---

## Когда расширять на другие паттерны

Если встречается похожая ситуация (большой набор данных + правила, требующие полного покрытия) - применять reconciliation loop проактивно, не дожидаясь явной просьбы пользователя.

По мере появления других воспроизводимых приёмов проверки результата работы агента - дописывать их в этот файл.

---

## Антипаттерны

- Смешивать сверку с написанием кода в одном проходе - теряется независимость проверки.
- Проверять выборочно вместо полного реестра.
- Останавливаться на «вроде работает» без явного нуля расхождений.
- Доверять имени/метке входа вместо его фактического содержимого.
- Считать сверку агента заменой приёмки человеком на реальных данных.
