# 1C Coding Standards

Стандарты написания кода на платформе 1С:Предприятие (BSL).

Применяется: при работе с кодом 1С (*.bsl, *.os).

Правила производительности (запросы в цикле, реквизиты через точку, кеширование), клиент-серверного взаимодействия (директивы компиляции, минимизация переходов, асинх-методы) и анти-паттерны описаны в `anti_patterns.md` и `form_module_rules.md` — здесь не дублируются.

---

## Комментарии

**НЕ добавляй комментарии по умолчанию.** Код должен быть понятен без комментариев. Комментарии допускаются **только** когда добавляют ценность, которую невозможно выразить самим кодом:

- Мотивация неочевидного решения (почему выбран именно такой подход)
- Нетривиальный алгоритм (суть, а не пересказ кода)
- Ограничения и граничные случаи
- Контракты экспортных функций (параметры, возвращаемое значение)
- Технический долг (TODO, FIXME)

Запрещено:
- Комментарии вида `// Получаем данные`, `// Проверяем условие`, `// Записываем результат` — они дублируют код
- Комментарии к каждой строке или блоку — код не нуждается в построчном пересказе
- Комментарии-разделители (`// ---`, `// ***`) вместо `#Область`

## Структура модулей

Порядок областей в общем модуле:

1. `#Область ПрограммныйИнтерфейс` — экспортные, доступны всем, документированы
2. `#Область СлужебныйПрограммныйИнтерфейс` — экспортные, доступны внутри подсистемы
3. `#Область СлужебныеПроцедурыИФункции` — неэкспортные, доступны внутри модуля

Документация экспортных функций:

```bsl
// Получает данные клиента
//
// Параметры:
//   Клиент - СправочникСсылка.Клиенты - ссылка на клиента
//
// Возвращаемое значение:
//   Структура - данные клиента с полями:
//     * Наименование - Строка
//     * ИНН - Строка
//
Функция ПолучитьДанныеКлиента(Клиент) Экспорт
```

## Именование

### Переменные

Не используй венгерскую нотацию (`МассивКонтрагентов` -> `Контрагенты`). Исключение: типы коллекций (`ТаблицаКлиентов`, `СписокДокументов`).

### Глобальный контекст

Не используй в именах переменных имена из глобального контекста 1С: `Документы`, `Справочники`, `РегистрыСведений`, `РегистрыНакопления`, `Метаданные`, `Константы`, `ПараметрыСеанса` и т.д.

```bsl
// Плохо:
Документы = Новый Массив;

// Хорошо:
СписокДокументов = Новый Массив;
```

### Процедуры и функции

- Процедуры (действие): `ЗаполнитьТаблицу()`, `ОбновитьДанные()`
- Функции-предикаты: `ЭтоНовый()`, `ЕстьОшибки()`, `МожноИзменить()`
- Функции (получение): `ПолучитьДанные()`, `НайтиКлиента()`, `ВычислитьСумму()`

## Запросы

### Русскоязычный синтаксис

В коде, планах, анализах и любых документах проектов 1С — использовать только русскоязычные ключевые слова языка запросов 1С. Англоязычные аналоги запрещены.

| Неправильно (англ.) | Правильно (рус.) |
|---------------------|------------------|
| `INNER JOIN` | `ВНУТРЕННЕЕ СОЕДИНЕНИЕ` |
| `LEFT JOIN` / `LEFT OUTER JOIN` | `ЛЕВОЕ СОЕДИНЕНИЕ` |
| `RIGHT JOIN` / `RIGHT OUTER JOIN` | `ПРАВОЕ СОЕДИНЕНИЕ` |
| `FULL JOIN` / `FULL OUTER JOIN` | `ПОЛНОЕ СОЕДИНЕНИЕ` |
| `CROSS JOIN` | `ДЕКАРТОВО ПРОИЗВЕДЕНИЕ` |
| `SELECT` | `ВЫБРАТЬ` |
| `FROM` | `ИЗ` |
| `WHERE` | `ГДЕ` |
| `GROUP BY` | `СГРУППИРОВАТЬ ПО` |
| `ORDER BY` | `УПОРЯДОЧИТЬ ПО` |
| `HAVING` | `ИМЕЮЩИЕ` |
| `UNION` / `UNION ALL` | `ОБЪЕДИНИТЬ` / `ОБЪЕДИНИТЬ ВСЕ` |
| `INTO` | `ПОМЕСТИТЬ` |
| `INDEX BY` | `ИНДЕКСИРОВАТЬ ПО` |
| `AS` | `КАК` |
| `ON` | `ПО` |
| `AND` / `OR` / `NOT` | `И` / `ИЛИ` / `НЕ` |
| `IN` | `В` |
| `BETWEEN` | `МЕЖДУ` |
| `LIKE` | `ПОДОБНО` |
| `IS NULL` | `ЕСТЬ NULL` |
| `CASE WHEN THEN END` | `ВЫБОР КОГДА ТОГДА КОНЕЦ` |
| `TRUE` / `FALSE` | `ИСТИНА` / `ЛОЖЬ` |
| `DISTINCT` | `РАЗЛИЧНЫЕ` |
| `TOP` | `ПЕРВЫЕ` |
| `ISNULL()` | `ЕСТЬNULL()` |

Это относится к: коду `.bsl`, текстам запросов в документации, примерам в планах и ТЗ, описаниям в анализах и ревью.

### Форматирование

Текст запроса без отступа на новой строке после `=`:

```bsl
Запрос.Текст =
"ВЫБРАТЬ
|    Таблица.Поле КАК Поле,
|    Таблица.Поле2 КАК Поле2
|ИЗ
|    Справочник.Таблица КАК Таблица
|ГДЕ
|    Таблица.Поле = &Параметр";
```

### Редактирование чужих запросов

При доработке существующего кода — сохраняй стиль оригинала:

1. **Не вводить конкатенацию строк** — если оригинальный запрос записан единым строковым литералом (без `+`), добавляемые строки вписываются внутрь этого литерала, а не через `+ "..."`.
2. **Сохранять выравнивание** — отступы `|`, позиции ключевых слов, стиль отступов оригинала.

```bsl
// Плохо: внесена конкатенация, хотя в оригинале её не было
Запрос.Текст =
    "ВЫБРАТЬ
    |    Таблица.Поле КАК Поле
    |ИЗ
    |    Справочник.Таблица КАК Таблица
    |ГДЕ
    |    Таблица.Условие1 = &Параметр1"
    // {Маркер начало}
    + "
    |    И Таблица.Условие2 = &Параметр2"
    // {Маркер конец}
    + "
    |    И Таблица.Условие3 = &Параметр3";

// Хорошо: новые строки вписаны в единый литерал, выравнивание сохранено
Запрос.Текст =
    "ВЫБРАТЬ
    |    Таблица.Поле КАК Поле
    |ИЗ
    |    Справочник.Таблица КАК Таблица
    |ГДЕ
    |    Таблица.Условие1 = &Параметр1
    // {Маркер начало}
    |    И Таблица.Условие2 = &Параметр2
    // {Маркер конец}
    |    И Таблица.Условие3 = &Параметр3";
```

Если оригинальный запрос уже использует конкатенацию — допустимо продолжить в том же стиле.

### РАЗРЕШЕННЫЕ - для всех запросов к реальным таблицам

Во всех запросах, обращающихся к **реальным** таблицам (справочники, документы, регистры, их виртуальные таблицы — `.СрезПоследних`, `.Остатки`, `.Обороты`), добавлять директиву `РАЗРЕШЕННЫЕ` после `ВЫБРАТЬ`.

Без `РАЗРЕШЕННЫЕ` при включённом RLS (Row Level Security) запрос падает с ошибкой "Недостаточно прав" для пользователей с ограниченным доступом, даже если сами данные им не нужны. `РАЗРЕШЕННЫЕ` заставляет платформу тихо отфильтровать строки, которые пользователь не имеет права видеть, а не падать.

Правило применяется к **каждому пакету** пакетного запроса отдельно.

Когда **НЕ** нужна директива:
- Пакет обращается **только** к временным таблицам (`ВТ...`)
- Запрос выполняется в привилегированном режиме (`УстановитьПривилегированныйРежим(Истина)`) — но это отдельная защита, и `РАЗРЕШЕННЫЕ` не повредит

```bsl
// ПРАВИЛЬНО: на каждом пакете с реальной таблицей
"ВЫБРАТЬ РАЗРЕШЕННЫЕ
|	Сотрудники.Ссылка КАК Ссылка
|ПОМЕСТИТЬ ВТСотрудники
|ИЗ
|	Справочник.Сотрудники КАК Сотрудники
|;
|
|// Этот пакет только по ВТ — РАЗРЕШЕННЫЕ не нужна
|ВЫБРАТЬ
|	ВТСотрудники.Ссылка КАК Сотрудник
|ИЗ
|	ВТСотрудники КАК ВТСотрудники
|;
|
|// Следующий пакет снова обращается к реальной таблице — опять РАЗРЕШЕННЫЕ
|ВЫБРАТЬ РАЗРЕШЕННЫЕ
|	Документы.Ссылка КАК Ссылка
|ИЗ
|	Документ.ПриемНаРаботу КАК Документы
|		ВНУТРЕННЕЕ СОЕДИНЕНИЕ ВТСотрудники КАК ВТСотрудники
|		ПО Документы.Сотрудник = ВТСотрудники.Ссылка"
```

Диагностика проблемы: запрос работает под админом, падает у обычного пользователя с "Недостаточно прав" — скорее всего, пропущена `РАЗРЕШЕННЫЕ`.

### Промежуточная переменная

Для результата запроса используй промежуточную переменную:

```bsl
// Плохо:
Выборка = Запрос.Выполнить().Выбрать();

// Хорошо:
РезультатЗапроса = Запрос.Выполнить();
Выборка = РезультатЗапроса.Выбрать();
```

### Псевдонимы полей

Всегда давай псевдонимы полям через `КАК`:

```bsl
ВЫБРАТЬ
    Контрагенты.ИНН КАК ИНН,
    Контрагенты.Наименование КАК Наименование
ИЗ
    Справочник.Контрагенты КАК Контрагенты
```

### Псевдонимы таблиц

Псевдоним таблицы должен быть информативным и понятным без контекста:

- Не использовать однобуквенные (`Р`, `Т`, `С`) или невнятные аббревиатуры (`ВТДок`, `ДокГР`)
- Для длинных имен таблиц - сокращать осмысленно: `РасчетыНДФЛ`, `РезидентствоФЛ`, `ВедомостиЗП`
- Для временных таблиц в роли источника - допустимо повторить имя ВТ или осмысленно сократить
- Псевдоним таблицы НЕ должен совпадать с именем поля (измерения, ресурса, реквизита) этой таблицы, используемого в запросе. Совпадение приводит к конструкции `Алиас.Алиас` — платформа не может разрешить имя и выдаёт ошибку

```bsl
// Плохо:
РегистрНакопления.РасчетыНалогоплательщиковСБюджетомПоНДФЛ КАК Р

// Хорошо:
РегистрНакопления.РасчетыНалогоплательщиковСБюджетомПоНДФЛ КАК РасчетыНДФЛ

// Плохо:
ВТДокументОснованияДляУвольнений КАК ВТУвольнения

// Хорошо:
ВТДокументОснованияДляУвольнений КАК ДокументыУвольнений

// Плохо: алиас "Местонахождение" совпадает с измерением регистра МестонахождениеОС
ВЫБРАТЬ
    Местонахождение.Местонахождение КАК Подразделение  // ошибка: двусмысленность имени
ИЗ
    РегистрСведений.МестонахождениеОС.СрезПоследних КАК Местонахождение

// Хорошо: алиас не пересекается с полями таблицы
ВЫБРАТЬ
    МестонахождениеОС.Местонахождение КАК Подразделение  // однозначно
ИЗ
    РегистрСведений.МестонахождениеОС.СрезПоследних КАК МестонахождениеОС
```

## Сообщения и ошибки

### Вывод сообщений

Не используй `Сообщить()`. Используй БСП-методы:

```bsl
// На сервере:
ОбщегоНазначения.СообщитьПользователю(
    "Ошибка при обработке данных", , "Объект.Реквизит", , Отказ);

// На клиенте:
ОбщегоНазначенияКлиент.СообщитьПользователю(
    "Ошибка при обработке данных");
```

### Обработка исключений

Не глотай исключения молча. Логируй и пробрасывай:

```bsl
Попытка
    // Опасная операция
Исключение
    ТекстОшибки = ПодробноеПредставлениеОшибки(ИнформацияОбОшибке());

    ЗаписьЖурналаРегистрации(
        "Модуль.Функция",
        УровеньЖурналаРегистрации.Ошибка, , , ТекстОшибки);

    ВызватьИсключение;
КонецПопытки;
```

## Даты и время

Для текущей даты на сервере используй `ТекущаяДатаСеанса()`, не `ТекущаяДата()`. `ТекущаяДата()` зависит от часового пояса сервера, `ТекущаяДатаСеанса()` учитывает часовой пояс пользователя.

## Конкурентность

Проверяй код на конкурентность и необходимость блокировок. Блокировки нужны при: изменении данных, чтении с последующей записью, работе с последовательностями/автонумерацией, критических секциях.

Предпочитай управляемые блокировки:

```bsl
Блокировка = Новый БлокировкаДанных;
ЭлементБлокировки = Блокировка.Добавить("Справочник.Клиенты");
ЭлементБлокировки.УстановитьЗначение("Ссылка", СсылкаКлиента);
Блокировка.Заблокировать();
```

Учитывай внешние транзакции: обработчики событий объектов (ПередЗаписью, ПриЗаписи) выполняются в транзакции записи — дополнительные блокировки не нужны.

## Форматирование

### Ограничение строки — 120 символов

Переносить строки длиннее 120 символов. Если строка укладывается в 120 символов — **не делать лишних переносов**, оставлять в одну строку.

```bsl
// Плохо (длинная строка >120 символов):
Результат = ОбщегоНазначения.ЗначенияРеквизитовОбъектов(МассивОбъектов, "Наименование, ИНН, КПП, ЮридическийАдрес, ФактическийАдрес");

// Хорошо (перенос):
Результат = ОбщегоНазначения.ЗначенияРеквизитовОбъектов(
    МассивОбъектов,
    "Наименование, ИНН, КПП, ЮридическийАдрес, ФактическийАдрес");

// Плохо (лишний перенос — строка укладывается в 120 символов):
ОбщегоНазначенияКлиент.СообщитьПользователю(
    "Ошибка при обработке данных");

// Хорошо (одна строка <=120 символов):
ОбщегоНазначенияКлиент.СообщитьПользователю("Ошибка при обработке данных");
```

### Выравнивание при заполнении объекта

При последовательном заполнении свойств объекта — выравнивать знаки `=` по самому длинному имени свойства. Ограничение 120 символов на строку сохраняется.

```bsl
// Плохо (не выровнено):
НастройкаОбъект.Наименование = "Формирование уведомлений по ПП";
НастройкаОбъект.ИмяДляРазработчика = "ФормированиеУведомлений";
НастройкаОбъект.СписокЗначений = Ложь;

// Хорошо (выровнено по =):
НастройкаОбъект.Наименование         = "Формирование уведомлений по ПП";
НастройкаОбъект.ИмяДляРазработчика   = "ФормированиеУведомлений";
НастройкаОбъект.СписокЗначений       = Ложь;
```

Аналогично для `Структура.Вставить`, `Соответствие.Вставить` и других однотипных блоков присвоения.

### Тернарный оператор

Не используй `?(Условие, Значение1, Значение2)`. Используй `Если...Тогда...Иначе`.

## Проверка пустоты коллекций

Используй `ЗначениеЗаполнено()` вместо `.Количество() = 0` для проверки пустоты коллекций (Массив, ТаблицаЗначений, Соответствие и др.):

```bsl
// Плохо:
Если МассивДанных.Количество() = 0 Тогда
Если МассивДанных.Количество() > 0 Тогда

// Хорошо:
Если НЕ ЗначениеЗаполнено(МассивДанных) Тогда
Если ЗначениеЗаполнено(МассивДанных) Тогда
```

---

## Булевы сравнения

Не сравнивай булевы выражения с `Истина`/`Ложь` - используй напрямую:

```bsl
// Плохо:
Если Условие = Истина Тогда
Если Флаг = Ложь Тогда

// Хорошо:
Если Условие Тогда
Если НЕ Флаг Тогда
```

## Операции с коллекциями

O(1) поиск через Соответствие, `ЗаполнитьЗначенияСвойств()`, `НайтиПоЗначению()` - подробнее и примеры: `anti_patterns.md` п.10.

## Привилегированный режим

Используй `УстановитьПривилегированныйРежим(Истина)` минимально и всегда выключай:

```bsl
УстановитьПривилегированныйРежим(Истина);
// Операция, требующая полных прав
УстановитьПривилегированныйРежим(Ложь);
```

Проверка текущего состояния: `ПривилегированныйРежим()`.

## Попытка/Исключение — ограничения

Не используй `Попытка...Исключение` для штатных операций с БД (чтение, запись), кроме обоснованного управления транзакциями. `Попытка` - для действительно опасных операций (COM, внешние вызовы, файлы).

### Транзакции — обязательный порядок

`НачатьТранзакцию()` - СТРОГО ДО `Попытка`, не внутри. Подробнее и шаблон: `anti_patterns.md` п.8.

- `ЗафиксироватьТранзакцию()` - последний оператор перед `Исключение`
- `ОтменитьТранзакцию()` - первый оператор в блоке `Исключение`
- Подробнее и примеры: `anti_patterns.md`, пункт 8

## Переиспользование кода

Перед написанием нового кода проверяй общие модули и модули менеджеров на наличие существующих экспортных методов. Не дублируй логику, которая уже реализована в конфигурации или БСП.

## Самоконтроль

После редактирования кода проверяй: стиль, читаемость, корректность логики, обработку ошибок, безопасность (нет SQL injection, проверены права), конкурентность (блокировки, гонки).

### Обязательные автоматические проверки

После каждого изменения BSL-кода — выполнить проверки performance и logic через MCP. Обязательны при доступности MCP-сервера. Если MCP недоступен — ручная проверка по `anti_patterns.md` + `code-review-checklist.md` (см. Fallback ниже).

**Шаг 1. Проверка доступности MCP**

Вызвать `get_edt_version` (1c-edt). Если MCP-сервер недоступен — сообщить пользователю:

> MCP-сервер 1c-edt недоступен. Для автоматической проверки performance и logic необходимо запустить EDT и MCP-сервер. Без MCP возможна только ручная проверка по anti_patterns.md и code-review-checklist.md.

При недоступности MCP — выполнить ручную проверку (шаг 5) и продолжить.

**Шаг 2. Performance — запросы**

- **ОБЯЗАТЕЛЬНО** `validate_query` — СРАЗУ после написания/изменения каждого запроса. Не откладывать до конца
- Проверить: синтаксис, семантика, для СКД — с `dcsMode=true`

**Шаг 3. Performance + Logic — ошибки проекта**

- `get_project_errors` с фильтром по изменённым объектам — ошибки производительности (запрос в цикле, обращение через точку, N+1), логические ошибки, типы, неиспользуемые переменные
- `revalidate_objects` — перевалидация изменённых объектов после исправлений

**Шаг 4. Качество кода - 1С:Напарник**

- **ОБЯЗАТЕЛЬНО** `ask_1c_ai` (1c-naparnik) - анализ стиля, архитектуры, соответствия стандартам 1С. Вызывать СРАЗУ после каждого написания/изменения BSL-кода. Не пропускать
- `check_1c_code` (1c-naparnik) - проверка синтаксиса, логики и производительности (запросы в цикле, N+1)
- `review_1c_code` (1c-naparnik) - code review по стандартам ИТС (именование, структура, читаемость). Перед коммитом
- `explain_1c_syntax` (1c-naparnik) - объяснение незнакомых конструкций BSL
- `check_syntax` (1c-syntax-checker-mcp) - альтернатива EDT для проверки синтаксиса

**Шаг 4б. Документация и стандарты - 1С:Напарник**

- `its_help` + `fetch_its` (1c-naparnik) - поиск и чтение стандартов/статей ИТС
- `search_1c_documentation` / `onec_help` (1c-naparnik) - документация API платформы (по версии / последняя)
- `config_help` (1c-naparnik) - документация конфигураций (ERP, ЗУП, Бухгалтерия)
- `diff_1c_documentation_versions` (1c-naparnik) - сравнение документации между версиями платформы

**Шаг 5. Fallback — ручная проверка (без MCP)**

Если MCP недоступен, проверить вручную по чек-листам:
- `anti_patterns.md` — запрос в цикле, реквизиты через точку, O(n²), клиент-серверные переходы
- `code-review-checklist.md` — логические ошибки, обработка исключений, безопасность

### Throttling

- `get_project_errors` / `check_syntax` - максимум 3 вызова за цикл редактирования
- `revalidate_objects` - максимум 2 вызова за цикл
- `ask_1c_ai` - максимум 2 вызова за цикл (отправить код, получить замечания, исправить, проверить повторно)
- После 2 неудачных исправлений одной и той же ошибки - остановиться, сообщить пользователю (вероятно нужен другой подход)

## Чистота diff

При редактировании существующего кода — каждая строка в diff должна быть обоснована задачей. Запрещено:

- Добавлять комментарии к коду, который не менялся функционально
- Добавлять комментарии к новому коду, если они просто пересказывают что делает код (см. секцию «Комментарии»)
- Добавлять `Попытка/Исключение` к коду, который работал без обработки ошибок и где нет нового риска
- Переименовывать переменные/функции в коде, который не менялся функционально
- Добавлять логирование (`ЗаписьЖурналаРегистрации`) если задача этого не требует
- Перемещать код между областями (`#Область`) без функциональной причины
- Переформатировать строки, которые не менялись функционально
